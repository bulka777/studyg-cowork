BM25 was published in 1994. The math is simple enough to fit on a whiteboard. Yet in production retrieval benchmarks in 2025, it still outperforms multi-billion-parameter dense embedding models on a meaningful slice of real-world queries. Teams that discover this after deploying pure vector search tend to discover it the worst possible way: through hallucination complaints they can't reproduce in evaluation, because their eval set was built from queries that already worked.

This is the retrieval equivalent of sampling bias. Dense retrieval fails on a specific and predictable query shape. The failure is silent — the LLM still produces fluent, confident-sounding answers from whatever fragments it retrieved. No error log fires. No latency spike. Just quietly wrong answers for users querying product SKUs, error codes, API names, or anything that is lexically specific rather than semantically general.

The fix is hybrid search. But "hybrid search" is underspecified as an engineering decision. This post covers what the failure modes actually look like, how to fuse retrieval signals correctly, where the reranking layer goes, and — most critically — how to find the query types your current pipeline is silently failing on before users find them for you.

The Queries Dense Retrieval Gets Wrong

Dense embedding models encode a passage into a single fixed-size vector by pooling over all token representations. This pooling is what enables semantic generalization — "vacation" and "time off" end up geometrically close — but it also destroys lexical identity for specific strings.

When a user queries ERR_SSL_VERSION_OR_CIPHER_MISMATCH, that exact token sequence gets averaged with every other token in the context window. The resulting vector captures something like "document about SSL errors" rather than "document containing this specific error string." BM25, by contrast, scores against an inverted index of exact tokens. It either finds the term or it doesn't. There is no blurring.

🔗 Concept link (lexical-search): This is exactly the mechanism your progress file records as shaky — "BM25 matches exact tokens only." The article now gives you the flip side: that precision is a feature, not just a limitation. BM25's inability to blur is what makes it reliable for identifiers.

The query types that systematically fail under pure dense retrieval follow a consistent pattern:

  • Error codes and identifiers: Exact strings like 0x80070005, INV-2024-00847, or ENOMEM have near-zero semantic representation signal.
  • Product SKUs and model numbers: RTX-4090 and RTX-4070 are semantically almost identical in embedding space — but they're different products.
  • Function names and library identifiers: Querying for torch.nn.functional.cross_entropy produces an embedding close to other PyTorch docs, not reliably to documents containing that specific call signature.
  • Rare named entities: When a query contains one rare and highly specific term alongside common words, the rare term's signal gets averaged away.
  • Domain jargon with controlled vocabulary: Legal, medical, and technical records use specific terms-of-art where semantic blurring introduces errors.

The 2021 BEIR benchmark made this embarrassingly clear: dense retrieval models trained on MS MARCO frequently failed to outperform BM25 in zero-shot cross-domain evaluation.

The Silent Failure Pattern

The harder problem is that these failures don't announce themselves. When BM25 misses a document, the retrieval set is visibly incomplete. When a dense retriever misses a document because the query contained a rare identifier, the LLM still gets a retrieval set — just the wrong one. It generates a fluent, plausible response from whatever fragments were retrieved. This is exactly the shape of hallucination that is hardest to catch in evaluation: confident, internally consistent, and wrong about specifics.

The diagnostic asymmetry compounds over time. Eval sets built from production query logs contain cases where the system produced answers users accepted. The cases where the system confidently provided wrong specifics are systematically absent from the feedback signal. You end up optimizing retrieval against the queries you already handle well.

Score Fusion: RRF vs. Convex Combination

Hybrid retrieval produces two ranked lists — one from BM25, one from dense ANN — and needs to merge them into a single ranked result.

Reciprocal Rank Fusion (RRF) converts each candidate document's position in each ranked list into a score: 1/(k + rank), where k=60 by convention. Documents that rank high in both lists accumulate higher scores. The key advantage is that RRF is score-scale agnostic — BM25 scores and cosine similarities are on completely incompatible scales, and RRF sidesteps normalization by operating on ranks rather than scores.

This makes RRF a safe default. It requires no labeled data, it's robust to distribution shift, and all major vector databases ship it natively: Elasticsearch 8.x, OpenSearch 2.12+, Weaviate, Qdrant.

Convex combination addresses the limitation of discarded score magnitude: score = α × score_dense + (1-α) × score_sparse. Bruch et al. (2022) showed that with as few as ~40 labeled query-relevance pairs, tuning this single α parameter consistently outperforms RRF.

In practice, α should vary by query domain:

  • Technical documentation with controlled terminology: α ≈ 0.3 (weight sparse heavily)
  • Conversational and policy documents: α ≈ 0.7–0.8 (weight semantic heavily)
  • Balanced mixed content: α ≈ 0.6

The Reranking Layer

Retrieval is a high-recall problem. Reranking is a high-precision problem. They require different models.

The structure is two-stage:

  1. First stage (retrieval): Hybrid BM25 + dense ANN with RRF fusion, fetching top-100 candidates. Fast, high-recall, operating on pre-computed indices.
  2. Second stage (reranking): A cross-encoder model scores each (query, candidate) pair jointly — the model sees both simultaneously and produces a relevance score with full attention between tokens. Applied to the top 30–50 candidates.

Cross-encoders are dramatically more accurate than bi-encoders because they can capture phrase-level alignment and inter-token dependencies. The cost is that they cannot pre-compute document representations. This makes them impractical as first-stage retrievers at any meaningful scale, but appropriate for reranking a small candidate set.

The recommended production pipeline: hybrid retrieval → top-100 → MMR de-duplication → cross-encoder reranking → top-5 to top-10 for the LLM context window.

Diagnosing Which Query Types You're Failing On

Stratify by query type. Cluster your query logs: exact identifier queries, keyword-heavy queries, semantic queries, navigational queries. Compute Recall@K separately for each cluster. If keyword and identifier clusters have 40% recall while semantic clusters have 85%, you know exactly where to invest.

Synthesize adversarial queries. Use an LLM to generate synthetic queries with exact identifiers drawn from your document corpus. Run these through your retrieval pipeline and check whether the source documents come back.

A production system with healthy hybrid retrieval should see Recall@10 around 85–91%, MRR above 0.80, and Hit Rate@10 above 90% for FAQ-style applications.

What to Actually Build

For most teams, the right starting point is RRF-based hybrid search using the native capabilities of their existing vector database. Once you have 40+ labeled query-relevance pairs for your domain, migrate from RRF to a tuned convex combination. Add cross-encoder reranking when you observe a persistent precision gap.

The highest-ROI diagnostic investment is building an adversarial eval set with exact-identifier queries before optimizing anything else. Dense embeddings are genuinely powerful for semantic retrieval. BM25 is genuinely powerful for lexical retrieval. It's now entirely about how precisely you understand where each one fails.

Historical context
1972 – SMART / TF-IDF
Salton's SMART system formalises term frequency × inverse document frequency as a retrieval score. Rare words that appear often in a document are more discriminative.
1994 – BM25 (Okapi BM25)
Robertson et al. refine TF-IDF with saturation (a document having a term 100× doesn't score 100× better) and length normalisation. Still the standard sparse baseline 30 years later.
2013 – Word2Vec
Mikolov et al. show that word meaning can be embedded in continuous vectors. "King − Man + Woman ≈ Queen." This is the seed of semantic retrieval.
2018–2019 – BERT bi-encoders (Dense Passage Retrieval)
Pre-trained transformers let you encode whole passages into vectors. DPR (Karpukhin et al., 2020) shows dense retrieval can beat BM25 on Wikipedia open-domain QA. Teams over-generalise this result.
2021 – BEIR benchmark
Thakur et al. test dense models across 18 diverse retrieval tasks. BM25 outperforms fine-tuned dense models on several out-of-domain datasets. The "dense is always better" narrative collapses.
2022–2026 – Hybrid + reranking becomes standard
Production RAG systems settle on: BM25 + dense ANN → RRF fusion → cross-encoder reranking. The article you just read is a production-hardened guide to this stack.
The mechanism beneath the thing

How BM25 actually works (lexical-search rubric)

Step 1: Inverted index

At index time, every document is tokenised. A dictionary maps each token → list of (document_id, position). Query token "SSL" → instantly retrieve all docs containing "SSL".

Step 2: BM25 score

score(D,Q) = Σ IDF(qᵢ) · (f(qᵢ,D)·(k₁+1)) / (f(qᵢ,D) + k₁·(1 - b + b·|D|/avgdl))

Where: IDF = log((N-n+0.5)/(n+0.5)), f = term freq in doc, k₁ ≈ 1.5, b ≈ 0.75 (length penalty).

Strength: rare terms

IDF is high for rare terms. If only 3 docs in 1M contain "ENOMEM", those 3 docs score very high for any query mentioning "ENOMEM". No blurring, no approximation.

Weakness: synonyms

Query: "time off". Doc contains only "vacation". BM25 score = 0. There is no token overlap, so no match — even though the meaning is identical.

How dense embeddings work (embeddings rubric — frontier concept)

Step 1: Encoder model

A transformer (e.g. BERT, sentence-transformers) reads the text and outputs a fixed-size vector (e.g. 768 or 1536 dimensions). The model was trained so that semantically similar texts produce nearby vectors.

Step 2: Vector index

All document vectors are pre-computed and stored in a vector store (Pinecone, Weaviate, Qdrant). At query time, the query is embedded and approximate nearest neighbours are found.

Strength: synonyms

"Vacation" and "time off" land near each other in embedding space because the training corpus showed them used in similar contexts. Cosine similarity ≈ 0.85.

Weakness: lexical identity

Querying "ERR_SSL_VERSION_OR_CIPHER_MISMATCH" produces a vector close to "SSL handshake documents" — not reliably the one doc containing that exact error string.
