Enterprises have largely internalized the first lesson of Retrieval-Augmented Generation (RAG): better retrieval reduces hallucinations. The second lesson is arriving fast: retrieval alone isn't "reasoning." In complex knowledge environments — policies that reference other policies, cases that depend on chains of events, systems where the answer is a relationship — not just a paragraph — classic "retrieve → stitch → generate" starts to plateau.

That is where GraphRAG (knowledge-graph–enhanced RAG) is gaining traction. Instead of treating your knowledge base as isolated chunks, GraphRAG attempts to model entities and their relationships, enabling multi-hop queries, cross-document linking, and more structured grounding.

Why GraphRAG

Traditional RAG excels when the user's question matches a single passage. It struggles when the answer requires:

  • Multi-hop reasoning (A relates to B which triggers C…)
  • Cross-document synthesis (policy + ticket + runbook + architecture note)
  • Entity-centric questions ("Which customers were impacted by services that depend on X?")
  • Global thematic questions ("What are the recurring root causes across incidents?")

Many GraphRAG approaches build a graph during indexing (entities/relations), then use it at query time via graph traversal, summarization, or both.

The 2026 GraphRAG Landscape

Microsoft GraphRAG — open-source, local vs. global query model

Microsoft Research's open GraphRAG pipeline builds a graph + community hierarchy and uses it to answer questions over private corpora. A well-documented separation between local search (entity-based context building) and global search (community-report map-reduce summarization for broad questions). Strong fit for Microsoft ecosystem (Azure OpenAI).

LightRAG (HKU) — dual-level retrieval

A research-to-open-source implementation combining graph structure with text indexing. A dual-level retrieval system covering both "low-level factual" and "high-level conceptual" queries in a unified approach.

fast-graphrag — lightweight, customizable

An open-source "speed-first" GraphRAG implementation using graph exploration techniques including Personalized PageRank at inference time. A toolkit for experimentation — not an end-to-end enterprise system.

EcphoryRAG — research-driven multi-hop

A newer research framework modeling retrieval as cue-driven activation and multi-hop expansion over an entity-centered graph. Claims average Exact Match improvement from 0.392 to 0.474 over selected KG-RAG baselines on multi-hop QA datasets.

A Simple Selection Map

  • Enterprise delivery: KnowCosmos Hybrid RAG on Galaxybase
  • Microsoft ecosystem + open framework: Microsoft GraphRAG
  • Research-backed open-source: LightRAG
  • Lightweight + hackable baseline: fast-graphrag
  • Multi-hop retrieval R&D: EcphoryRAG

Section 2 — Under the Hood: How GraphRAG Actually Works

Section 1 above covers the annotated article. This section opens GraphRAG up end-to-end — how text is chunked and vectorized, what the three vector stores actually contain, and exactly how a query is routed between the entity store and the community store. Grounded primarily in the Microsoft GraphRAG indexing & query documentation; the green/blue seam convention from above still applies.

1 · The Indexing Pipeline — Six Phases from Raw Text to Knowledge Model

Sourced from Microsoft GraphRAG "Default Dataflow" documentation. This is the build step — it runs once (and on updates), not per query.
The default GraphRAG indexing dataflow. Only three artifacts get embedded (Phase 6) — everything else is traversed structurally.
1
Compose Text Units (chunking)
Each document is split into Text Units — chunks of a configurable token count, 1200 tokens by default. Each text unit gets a unique ID and keeps a pointer back to its source document, so every fact extracted later can be traced to its origin (provenance / "breadcrumbs"). Larger chunks process faster but give lower-fidelity extraction and less precise references.
2
Document Processing
A Documents table is built that links each document to its constituent text units. This is bookkeeping for provenance — documents themselves are not used directly downstream.
3
Graph Extraction — entities, relationships, (optional) claims
For each text unit, an LLM extracts entities (each with a title, type, description) and relationships (each with source, target, description). This produces one subgraph per text unit. The subgraphs are then merged: entities sharing the same title+type are collapsed into one node (their descriptions collected into a list); relationships sharing the same source+target are likewise collapsed. A second LLM pass — summarization — compresses each node's and edge's list of descriptions into a single concise description. Optionally, a separate pass extracts Claims (Covariates): time-bound factual statements about entities. Claims are off by default (they need prompt tuning to be useful).
4
Graph Augmentation — community detection
The merged entity-relationship graph is clustered using the Hierarchical Leiden algorithm. This recursively partitions the graph into communities (densely-connected clusters of entities) until a size threshold is reached, producing a hierarchy: level 0 = a few large communities covering the whole corpus; deeper levels = many small, tightly-scoped communities. This hierarchy is what later lets you answer questions at different "zoom levels."
5
Community Summarization — community reports
For every community at every level, an LLM writes a community report: an executive summary plus the key entities, relationships, and claims inside that cluster. A top-level community yields a report about the whole dataset; a low-level community yields a report about one tight cluster. A shorthand summary of each report is also generated.
6
Text Embedding — populate the vector stores
As the final step, the artifacts that need semantic search are embedded and written to a vector store. By default exactly three things are embedded: (a) entity descriptions, (b) text-unit text, (c) community-report content. Relationships and covariates are not embedded — they are reached by graph traversal from entities, not by vector similarity.
Key correction to the simplified panel above: GraphRAG does not embed "the graph." It embeds three text artifacts derived from the graph. The graph's edges (relationships) are followed structurally — like SQL joins — not searched by similarity.

2 · Chunking & Vectorization — What Becomes a Vector, and What Doesn't

This is the part the buyer's-guide article skips entirely. It connects GraphRAG back to the chunking and embeddings concepts already on your map.
Chunking in GraphRAG is the same chunking you already know from basic RAG: documents → fixed-size token chunks (default 1200) → each chunk is a "Text Unit." The twist is that text units serve two roles: they are (1) the input fed to the LLM for entity/relationship extraction, and (2) embedded directly so they can still be retrieved as raw passages at query time. (Source: MS GraphRAG dataflow, Phase 1 & Phase 6.)
Walk the vectorization through one sentence. Suppose a text unit contains: "After the March 12 outage, the Auth Service team migrated off the legacy DB cluster."
  • As a text-unit vector: the whole sentence (within its 1200-token chunk) is embedded as-is → 1 vector in the text-unit store.
  • As entities: the LLM pulls out Auth Service (type: service) and legacy DB cluster (type: database), each with a generated description. Those descriptions — not the raw sentence — are what get embedded into the entity store.
  • As a relationship: Auth Service —[migrated off]→ legacy DB cluster is stored as a graph edge. It is not embedded; it's found by traversing from either entity.

So the same source sentence is represented three different ways for three different retrieval mechanisms. This is the core insight behind "what's in the vector stores" — each store holds a different abstraction of the corpus: raw passages, distilled entity meaning, and whole-cluster themes.

3 · The Three Vector Stores — What Each One Holds

You were right that there are three. In default Microsoft GraphRAG these are three collections written in Phase 6. Each indexes a different artifact and is consulted by a different query path.
① Entity Description Store
default-entity-description
  • One vector per entity in the graph.
  • Embeds the summarized description of the entity (from Phase 3), not the raw text.
  • Vector links back to the entity node (id, title, type, degree).
Used as the entry point for Local Search & DRIFT — "which entities does this query mention?"
② Text-Unit Store
default-text-unit-text
  • One vector per chunk (~1200 tokens).
  • Embeds the raw chunk text verbatim.
  • This is the "classic RAG" store — identical in spirit to a normal vector DB.
Supplies raw evidence passages — reached in Local Search via entity→text-unit mapping, and used directly by Basic (vanilla) Search.
③ Community Report Store
default-community-full-content
  • One vector per community report, across all hierarchy levels.
  • Embeds the full report content (LLM-written cluster summary).
  • Holds whole-theme / "big picture" abstractions.
The "global" store — used by DRIFT's primer phase and dynamic global search to find thematically relevant clusters.
"For all artifacts that require downstream vector search, we generate text embeddings… By default we embed entity descriptions, text unit text, and community report text." — Microsoft GraphRAG, Indexing Dataflow, Phase 6.
What is NOT in any vector store: relationships (edges) and covariates/claims. These carry no embedding by default — they are retrieved structurally, by following the graph out from entities that vector search already found. That division — vectors to enter the graph, edges to walk it — is the whole trick.

4 · Query Routing — Entity Store vs. Community Store

"When do we go to the community store vs. the entity store?" GraphRAG ships four query modes. The mode determines which store is hit first and how the graph is walked. Sourced from MS GraphRAG Query Engine docs.
ModeFirst store hitMechanismBest for
Local Search ① Entity store Embed query → nearest entities → fan out via graph maps to their text units, relationships, neighbor entities, covariates, and the community reports they belong to → rank/filter to one context window. Questions about specific entities ("What are chamomile's healing properties?").
Global Search ③ Community store No vector search by default — takes all community reports at a chosen hierarchy level, map-reduce: each batch yields rated points (0–100), then top points are aggregated into the answer. Whole-corpus / thematic questions ("What are the top 5 themes?").
DRIFT Search ③ then ① Primer: match query against community reports → seed answer + follow-up questions. Then run Local Search on each follow-up. Then combine. Blends global breadth with local precision. Complex questions needing both context and specifics.
Basic Search ② Text-unit store Plain vector RAG: embed query → top-k nearest text-unit chunks → stuff into prompt. Provided as a baseline for comparison. Single-passage lookups; sanity-checking vs. graph modes.

Local Search — the entity store as a doorway into the graph

Local Search: vector search hits the entity store once to find access points, then the rest is graph traversal — pulling in text units, reports, neighbors, edges and claims, all ranked to fit one context window. (Recreated from the MS Local Search dataflow.)

Global Search — the community store, no entity lookup at all

Global Search: skips entity lookup entirely. It map-reduces over community reports at a hierarchy level you choose — lower levels = more detail but more cost. (Recreated from the MS Global Search dataflow.)

Worked routing examples — same corpus, three questions

"What are the healing properties of chamomile?"
  1. Route → Local. The question names a specific entity (chamomile).
  2. Query embedded → matched against the entity store ① → finds the chamomile node.
  3. Fan out: its text units (② raw passages mentioning chamomile), its relationships (chamomile —[treats]→ insomnia), neighbor entities, and the community reports it belongs to (③).
  4. Rank, fit one context window, generate. Entity store leads; community store is only a supporting input.
"What are the top 5 recurring themes across all these herbal notes?"
  1. Route → Global. "Across all" = whole-corpus synthesis; no single entity to anchor on.
  2. Skip the entity store entirely. Take all community reports ③ at a chosen level.
  3. Map: each batch of reports → rated candidate themes. Reduce: keep the highest-rated → final answer.
  4. Community store is the whole show; entity store is never touched.
"How did the chamomile supply shortage affect the products that depend on it?"
  1. Route → DRIFT. Needs broad context (what's the shortage about) and specific multi-hop chains (which products, which dependencies).
  2. Primer: match query against community reports ③ → seed answer + follow-up questions.
  3. Follow-ups run as Local searches off the entity store ①, walking chamomile →[ingredient-of]→ product edges.
  4. Combine global framing with local specifics into the final answer.

5 · Why This Is Expensive — and When It's Worth It

Ties the mechanism back to the buyer's-guide framing in the left panel.

Every phase 3–5 step is an LLM call per text unit or per community: extraction, description summarization, and one report per community at every hierarchy level. A corpus that costs cents to index for basic RAG can cost orders of magnitude more for GraphRAG, because you are paying an LLM to read everything several times during indexing. That is the indexing-cost line in the comparison table above made concrete.

The payoff is at query time: multi-hop and whole-corpus questions that basic RAG simply cannot answer become tractable, because the expensive reasoning was front-loaded into the index (entities, edges, and pre-written community summaries) instead of being attempted on-the-fly from a handful of similar chunks. The practical 2026 guidance from the source article holds: use GraphRAG where the answer is a relationship or a theme, and keep plain vector RAG for single-passage lookups.
